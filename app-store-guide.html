7. App Privacy (Data Collection Labels)

Critical: Apple requires you to accurately declare all data collection in the App Privacy section. Incorrect declarations can cause rejection.

Data Types to Declare

Data TypeCollectionUsageLinked to UserTracking
Health & Fitness Yes App Functionality No (on-device only) No
Precise Location Yes App Functionality No (on-device only) No
Diagnostics (Crash Data) Yes App Functionality No No
Usage Data (Analytics) Yes Analytics No No
Purchase History Yes App Functionality No No

Step-by-Step Privacy Questionnaire

  1. Go to your app in App Store Connect → App Privacy
  2. Click Get Started
  3. "Does your app collect data?" → Yes
  4. Select data types:
    • Health & Fitness → Health (HealthKit data)
    • Location → Precise Location
    • Diagnostics → Crash Data, Performance Data
    • Usage Data → Product Interaction
    • Purchases → Purchase History
  5. For each data type, answer:
    • Used for tracking? No
    • Linked to user's identity? No
    • Purposes: App Functionality / Analytics
  6. Click Publish
Note: Water Reminder Tracker App does NOT use ATT (App Tracking Transparency) because no third-party tracking or advertising is performed. Firebase Analytics is configured for first-party analytics only.

10. Archive & Upload from Xcode

Pre-Archive Checklist

  • Version is set to 1.0.1 and build to 3 (or next number)
  • Scheme is set to Release (not Debug)
  • Device is set to "Any iOS Device (arm64)"
  • All capabilities are enabled (HealthKit, WeatherKit, Push, App Groups, IAP)
  • GoogleService-Info.plist is included in the bundle
  • No compiler warnings or errors
  • App icon is 1024x1024px, no alpha, 72 DPI, sRGB

Steps to Archive

  1. In Xcode, select Product → Archive (or Cmd + Shift + B to build first)
  2. Wait for the archive to complete (may take 2-5 minutes)
  3. The Organizer window opens automatically showing your archive
  4. Click Distribute App
  5. Select App Store Connect → Upload
  6. Keep defaults:
    • Upload your app's symbols: Yes
    • Manage version and build number: Yes
  7. Select your Distribution Certificate and Provisioning Profile
  8. Click Upload
  9. Wait for upload + processing (5-15 minutes)
After Upload: Go to App Store Connect → your app → TestFlight tab. The build should appear within 15-30 minutes after processing completes. You'll receive an email when it's ready.

Final Submission Checklist

  • All app metadata is filled in (name, subtitle, description, keywords)
  • Screenshots uploaded for required device sizes
  • App icon uploaded (1024x1024, delivered in the binary)
  • Privacy Policy URL is accessible and up-to-date

Steps to Submit

  1. In App Store Connect, go to your app → latest version
  2. Verify all sections show green checkmarks
  3. Select the uploaded Build
  4. Set Release: "Manually release this version" (recommended for first launch)
  5. Click Submit for Review
  6. Confirm the submission

Review Timeline

  • Most apps are reviewed within 24-48 hours
  • First-time submissions may take longer (up to 5 days)
  • Check status at App Store Connect or via email notifications
  • If rejected, fix the issues and resubmit (no need to re-upload the binary if it's metadata-only)

14. Common Rejection Reasons & Fixes

1. Guideline 5.1.1 - Data Collection and Storage (HealthKit)

Issue: HealthKit data is used for purposes beyond app functionality, or privacy policy doesn't adequately cover HealthKit.

Fix: Ensure privacy policy explicitly states HealthKit data is never shared with third parties, never used for advertising, and is stored only on-device. Usage descriptions in Info.plist must be specific and accurate.

2. Guideline 3.1.2 - Subscriptions

Issue: Subscription terms not clearly communicated before purchase.

Fix: Paywall must show: price, duration, trial length, auto-renewal info, and links to Terms/Privacy. Our PaywallView already includes all of these.

3. Guideline 2.1 - App Completeness

Issue: App crashes or has obvious bugs during review.

Fix: Thoroughly test on real devices. Make sure all features work without a subscription (free features must be functional).

4. Guideline 5.1.2 - Data Use and Sharing (Location)

Issue: Location usage description is too vague.

Fix: Info.plist descriptions clearly explain WHY location is needed and that data stays on-device. Our descriptions are already specific.

5. Guideline 2.3.7 - Accurate Metadata

Issue: Screenshots don't match actual app UI.

Fix: Capture fresh screenshots from the latest build. Don't use mockups that differ from the real UI.

6. Missing In-App Purchase Metadata

Issue: IAP products show "Missing Metadata" status.

Fix: Each IAP needs: display name, description, review screenshot, and pricing configured. Check all products in the Subscriptions tab.
